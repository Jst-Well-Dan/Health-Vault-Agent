## 可行性分析

**不推荐部署到 GitHub Pages**，原因有两个核心问题：

### 1. 后端无法运行
GitHub Pages 只能托管纯静态文件，你的应用依赖：
- FastAPI 服务器（Python 运行时）
- SQLite 数据库（`data/health.db`）
- REST API（`/api/members` 等）

这些在 GitHub Pages 上完全无法运行。

### 2. 密码保护在静态站点上是假安全
GitHub Pages 的"密码保护"只能用 JS 实现，**任何人只要查看源码或绕过 JS 就能访问**，对医疗健康数据来说完全不可接受。

---

## 推荐替代方案

| 方案                       | 成本       | 难度 | 说明                                              |
| -------------------------- | ---------- | ---- | ------------------------------------------------- |
| **Railway / Render**       | 免费层可用 | 低   | 支持 FastAPI + SQLite，可加 HTTP Basic Auth       |
| **Fly.io**                 | 免费层可用 | 中   | 支持持久化存储，适合 SQLite                       |
| **本地局域网 + Tailscale** | 免费       | 低   | 只在你的设备上跑，通过 Tailscale 远程访问，最安全 |
| **VPS（轻量云服务器）**    | ~¥50/月    | 中   | 完全控制，可配 Nginx Basic Auth                   |

---

**我的建议**：用 **Tailscale** 最简单且最安全——本地运行后端，Tailscale 组网后手机/电脑都能访问，数据不出本地，零泄露风险。想了解哪个方案的具体步骤？
